===============================================================================
                    ЗАЩИТА ПРОЕКТА NeoMarket
===============================================================================

ВВЕДЕНИЕ
-------------------------------------------------------------------------------
Название проекта: NeoMarket — современная платформа электронной коммерции
                  для продажи электроники и бытовой техники.

Тип проекта: Full-stack веб-приложение (Frontend + Backend API)

Стек технологий:
  • Frontend: React 19, TypeScript, Vite, React Bootstrap 5
  • Backend: Microservices (B2B и B2C сервисы)
  • Архитектура: Component-based SPA

===============================================================================
1. АРХИТЕКТУРА ПРОЕКТА
===============================================================================

1.1 Общая структура
-------------------------------------------------------------------------------
Проект разделён на три основных модуля:

  • neomarket-frontend     — React SPA приложение для клиентов
  • neomarket-b2b-service  — Сервис для партнёров и оптовых покупателей
  • neomarket-b2c-service  — Сервис для розничных клиентов

1.2 Frontend архитектура
-------------------------------------------------------------------------------
src/
├── components/    # Переиспользуемые UI компоненты
│   ├── Header.tsx       # Шапка сайта с навигацией
│   ├── HeroSection.tsx  # Главный баннер
│   ├── ProductCard.tsx  # Карточка товара
│   └── SearchBar.tsx    # Поиск с автодополнением
├── pages/         # Страницы приложения (роуты)
│   ├── HomePage.tsx       # Главная страница
│   ├── CatalogPage.tsx    # Каталог товаров
│   ├── ProductPage.tsx    # Детали товара
│   ├── CartPage.tsx       # Корзина
│   ├── CheckoutPage.tsx   # Оформление заказа
│   ├── WishlistPage.tsx   # Избранное
│   ├── OrdersPage.tsx     # История заказов
│   └── AboutPage.tsx      # О проекте
├── services/      # Работа с API (axios clients)
│   ├── api.ts       # Основной API клиент
│   └── mockData.ts  # Моковые данные для разработки
├── context/       # React Context (глобальное состояние)
│   └── CartContext.tsx  # Управление корзиной
├── types/         # TypeScript интерфейсы и типы
│   └── index.ts
└── assets/        # Статические ресурсы

Принципы архитектуры:
  • Component-based — всё построено на переиспользуемых компонентах
  • Separation of Concerns — чёткое разделение логики, UI и данных
  • Type Safety — полная типизация через TypeScript

2. КЛЮЧЕВЫЕ ТЕХНОЛОГИИ И ПОЧЕМУ ОНИ ВЫБРАНЫ
===============================================================================

2.1 React 19
-------------------------------------------------------------------------------
Почему: Самая популярная библиотека для SPA, огромная экосистема,
        виртуальный DOM для производительности.
Преимущества: Компонентная архитектура, хуки, React Router для навигации.

2.2 TypeScript
-------------------------------------------------------------------------------
Почему: Статическая типизация предотвращает ошибки на этапе разработки.
Преимущества: IntelliSense, рефакторинг, самодокументирование кода.

2.3 Vite
-------------------------------------------------------------------------------
Почему: Мгновенный запуск dev-сервера, быстрая сборка, HMR.
Преимущества: В 10-100x быстрее Webpack, нативная поддержка ES Modules.

2.4 React Bootstrap 5
-------------------------------------------------------------------------------
Почему: Готовые компоненты, адаптивный дизайн, Material Design.
Преимущества: Не нужно писать CSS с нуля, кроссбраузерность.

2.5 Axios
-------------------------------------------------------------------------------
Почему: Простой HTTP-клиент с Promise API, перехватчиками.
Преимущества: Легче чем fetch, встроенная обработка ошибок.

5.2 Переключение между моковыми данными и реальным API
-------------------------------------------------------------------------------
const USE_MOCK_DATA = true; // false для продакшена

POST   /wishlist?user_id=:id            — Добавить в избранное
DELETE /wishlist/:id?user_id=:id        — Удалить из избранного

6.1 React Context
-------------------------------------------------------------------------------
Используется для глобального состояния (корзина, избранное):

// src/context/CartContext.tsx
export const CartContext = createContext<CartContextType | undefined>(undefined);

export function CartProvider({ children }: { children: ReactNode }) {
  const [cart, setCart] = useState<Cart | null>(null);
  
  const addToCart = async (item: CartItemCreate) => {
    const updatedCart = await cartService.addToCart(userId, item);
    setCart(updatedCart);
  };
  
  return (
    <CartContext.Provider value={{ cart, addToCart, removeFromCart }}>
      {children}
    </CartContext.Provider>
  );
}

Почему Context, а не Redux: Для небольших проектов Context проще и не
                            требует дополнительных зависимостей.

6.2 Использование Context в компонентах
-------------------------------------------------------------------------------
import { useCart } from '../context/CartContext';

function Header() {
  const { cart } = useCart();
  const cartItemCount = cart?.total_items || 0;
  
  return (
    <Navbar>
      {/* ... */}
      {cartItemCount > 0 && (
        <Badge>{cartItemCount}</Badge>
      )}
    </Navbar>
  );
}

15. ОТВЕТЫ НА ВОЗМОЖНЫЕ ВОПРОСЫ
===============================================================================

Q: Почему React, а не Vue/Angular?
-------------------------------------------------------------------------------
A: React — самая популярная библиотека, огромная экосистема, легче найти
   разработчиков и готовые решения.

Q: Почему TypeScript, а не JavaScript?
-------------------------------------------------------------------------------
A: Статическая типизация предотвращает ошибки на этапе разработки,
   улучшает IDE поддержку и документацию кода.

Q: Почему Vite, а не Create React App?
-------------------------------------------------------------------------------
A: Vite в 10-100x быстрее, использует нативные ES Modules, современный
   подход к сборке.

Q: Почему Context, а не Redux/Zustand?
-------------------------------------------------------------------------------
A: Для небольших проектов Context проще, не требует дополнительных
   зависимостей и boilerplate кода.

Q: Как будет работать аутентификация?
-------------------------------------------------------------------------------
A: JWT токены, хранящиеся в httpOnly cookies или localStorage, добавление
   interceptor в axios для автоматической подстановки токена.

Q: Как будет реализован SSR для SEO?
-------------------------------------------------------------------------------
A: Переход на Next.js или использование react-dom/server для
   предрендеринга критических страниц.

Q: Как обрабатываются ошибки API?
-------------------------------------------------------------------------------
A: try/catch блоки в сервисах, обработка ошибок в компонентах с отображением
   пользователю.

Q: Как масштабируется проект?
-------------------------------------------------------------------------------
A: Модульная структура, микросервисная архитектура бэкенда, возможность
   горизонтального масштабирования.
